Moving between Stata and R

A short guide to import/export, labelled data, dates, defaults, and safe cross-language workflows.

On this page:
  • A quick mental model
  • Import and export
  • General differences in practice
  • Data structures and how to unite them
  • Differences in defaults that can change results
  • Rules for splitting work across the two languages

If you move between Stata and R, the big challenge is rarely syntax alone. The real issue is that the two languages make different assumptions about data structures, metadata, and defaults.

This note is about moving safely between them without silently changing your data or results.

Import and export

If you want to preserve as much Stata metadata as possible, a Stata file is usually the safest handoff format.

GoalRecommended formatWhy
Move data between Stata and R while keeping labels.dtaKeeps variable labels and value labels better than plain text
Share a simple rectangular file.csvPortable, but loses labels, formats, and richer metadata
Store a frozen analysis dataset for R-heavy work.rdsBest for R-only work, but Stata cannot read it
 
use "analysis_data.dta", clear
export delimited using "analysis_data.csv", replace
import delimited using "analysis_data.csv", clear
save "analysis_data.dta", replace
library(haven)
library(readr)

df <- read_dta("analysis_data.dta")
write_dta(df, "analysis_data.dta")

df_csv <- read_csv("analysis_data.csv")
write_csv(df_csv, "analysis_data.csv")
Rule of thumb: If labels matter, prefer .dta. If you use .csv, assume you must rebuild labels, date formats, and missing-value logic explicitly.

General differences in practice

1. Loops are not the first tool in R

Stata users often reach for a loop early. In R, loops are valid, but many tasks are easier with vectorised code or with verbs such as mutate(), summarise(), and across().

foreach var of varlist bmi sbp dbp {
    replace `var' = . if `var' < 0
}
df |>
  dplyr::mutate(
    dplyr::across(c(bmi, sbp, dbp), \(x) dplyr::if_else(x < 0, NA_real_, x))
  )

2. R encourages reusable objects

In Stata, you often transform the active dataset in place. In R, it is common to keep several versions around: raw_data, clean_data, analysis_data, and plot_data.

3. Packages matter more in R

In Stata, most workflows feel like one integrated environment. In R, package choice matters a great deal: {haven}, {dplyr}, {labelled}, {lubridate}, {ggplot2}, and others.

Factor variables and labels

After haven::read_dta(), a categorical Stata variable often arrives as a labelled vector rather than a plain factor. That preserves codes and labels, but many models and plots behave more naturally with factors.

library(haven)

x <- read_dta("analysis_data.dta")
str(x$sex)

x$sex_factor <- as_factor(x$sex)
  • keep as labelled if you still need the original numeric codes
  • convert with haven::as_factor() if you want category labels used in models, tables, or plots
  • remove labels deliberately if the variable is actually continuous metadata with labels attached by habit

Missing values

Stata has multiple extended missing values, and they sort above ordinary numbers. R uses NA, and comparisons with NA usually return NA, not TRUE or FALSE.

Dates and origins

  • Stata daily dates: days since 1960-01-01
  • R Date: days since 1970-01-01
  • Stata datetimes: usually milliseconds since 1960-01-01 00:00:00
  • R POSIXct: usually seconds since 1970-01-01 00:00:00 UTC
as.Date(stata_days, origin = "1960-01-01")
as.POSIXct(stata_ms / 1000, origin = "1960-01-01", tz = "UTC")
Never trust raw imported date numbers. Always check a few known dates after import/export.

Differences in defaults that can change results

The dangerous cases are the ones where both programs run successfully but answer slightly different questions.

  • Factor/reference handling: in R, coefficients depend on the current factor coding and reference level.
  • Missing-data handling: different R functions and packages can use different missing-data rules.
  • Joins and merges: R joins are concise, but duplicate keys can silently expand rows.
  • Model fitting defaults: be explicit about robust or clustered SEs, weights, starting values, optimizer settings, and convergence tolerances when matching Stata and R.

Rules for splitting work across the two languages

  1. Pick one canonical source file for each stage.
  2. Do not hand-edit the same dataset in both languages.
  3. Keep identifiers stable.
  4. Freeze variable names before handoff.
  5. Export metadata intentionally.
  6. Check the handoff with a tiny audit.
  7. Write scripts, not manual clicks.
  8. Decide where models are fitted and where plots are made.

A safe bridge is usually: clean in one language, export an analysis-ready dataset, import in the other language, verify dimensions/missingness/labels/dates, and only then start modelling or plotting.
